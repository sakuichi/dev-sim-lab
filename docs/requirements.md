# 引き継ぎ書: ソフトウェア開発シミュレーターシリーズサイト構築

## 1. プロジェクト概要

エンジニアには常識だが経営層やステークホルダーには理解されにくいソフトウェア開発の問題を、インタラクティブに可視化する **One Issue、One Webサイトのシリーズ** を構築する。

- **起点**: QCDトレードオフシミュレーター(公開済み: `https://sakuichi.github.io/qcd-tradeoff/`)
- **用途**: 教育ツール、顧客・ステークホルダーへのトレーニング素材、Amazonアフィリエイトによるマネタイズ
- **ターゲット**: 日本語・英語の両言語ユーザー

## 2. 決定事項サマリ

| 項目 | 決定内容 |
|---|---|
| リポジトリ構成 | **1リポジトリに全シミュレーターを統合**(サブフォルダ方式) |
| QCDシミュレーター | **統合リポジトリに移設**(旧URLにリダイレクトを設置) |
| 初期実装スコープ | **ランディングページ + シミュレーター2〜3本**(ブルックスの法則、稼働率100%の罠を必須、余力があればコンテキストスイッチ) |
| 言語対応 | **最初から日英対応**(QCDのi18n方式を踏襲) |
| ホスティング | GitHub Pages(プロジェクトサイトが完全無料、独自ドメインなし) |
| 技術スタック | 素のHTML + CSS + JS、ビルド依存なし、ライブラリはCDN経由。ページ数が10を超えたらAstro/Eleventy移行を検討 |
| 連続画面 | 採用しない(クローンリスクと運用負荷軽減を優先) |
| 連絡先 | Googleフォーム(QCDと共通) |

## 3. リポジトリ構造(案)

リポジトリ名は `dev-sim-lab` に確定(URL: `sakuichi.github.io/dev-sim-lab/`)。

```
dev-sim-lab/
├── index.html              # 全シミュレーター一覧(ルート一覧)
├── qcd-tradeoff/
│   └── index.html          # 既存QCDシミュレーターを移設
├── brooks-law/
│   └── index.html          # ブルックスの法則
├── utilization-trap/
│   └── index.html          # 稼働率100%の罠
├── context-switch/
│   └── index.html          # コンテキストスイッチ(任意・3本目)
├── assets/
│   ├── common.css          # 共通デザイントークン・レイアウト
│   ├── common.js           # i18n基盤・シェアボタン・共通UI
│   └── ogp/                # OGP画像(Pillowで生成、シミュレーターごと)
├── privacy/
│   └── index.html          # プライバシーポリシー&免責事項(日英、QCDから流用)
├── sitemap.xml
├── robots.txt
└── docs/
    └── requirements.md     # 本要件定義(継続更新)
```

- 各シミュレーターは独立ページ(SPA不要)。1ページ = 1フォルダで追加していく。
- 共通化するもの: ヘッダー/フッター、言語切替、シェアボタン、カードスタイル、アフィリエイト枠。
- 共通化の実装は `common.js` に寄せるが、各ページ単体でも開ける構成を保つ(file://で開いて動作確認できること)。

## 4. 全ページ共通仕様

### 4.1 ページ構成(3部構成に統一)

すべてのシミュレーターページは以下の順で構成する:

1. **よくある誤解** — 経営層が抱えがちな誤認を冒頭の問いで提示(例:「人が増えれば早く終わる?」)
2. **シミュレーターで体験** — インタラクティブ操作で誤解が崩れる体験
3. **現実の対策** — 実務でどう向き合うかの提言

ページ末尾に「関連する問題」として他シミュレーターへの相互リンクを置く。

### 4.2 i18n(QCD方式を踏襲)

- `I18N = {ja, en}` の辞書オブジェクト + `data-i18n` 属性でのテキスト差し替え
- 言語切替UIはヘッダー右上、選択は `localStorage` に保存
- URLクエリ `?lang=en` でも切替可能にする(シェアリンク用)
- **開発者モード(🔥)**: QCDと同様、日英両対応の風刺的代替コピーを `DEV` オーバーレイ辞書で提供。通常コピーはステークホルダー中立(リスクを可能性として記述し誹謗を避ける)。開発者モードは意図的にその逆
- 辞書キー構造は全シミュレーターで揃える(`title`, `misconception`, `simulator.*`, `advice.*`, `share.*` など)

### 4.3 操作ポリシー

- 通常表示: 「〜になる可能性があります」など、特定の人物・組織を誹謗しない表現
- 開発者モード: 風刺OK。ただし実在の企業・人物名は出さない

### 4.4 マネタイズ・シェア

- **Amazonアフィリエイト**: JAは amazon.co.jp(既存タグを流用)、EN側は Amazon Associates US で実証されている対応(リンク枠だけ用意し、EN時は非表示または関連書籍紹介のみに切替)
- 各シミュレーターのテーマに沿った書籍を1〜3冊紹介(例: ブルックスの法則→「人月の神話」)
- **シェアボタン**: JA = X / はてなブックマーク / Facebook、EN = X / LinkedIn / Facebook。LINEは不採用(ビジネス文脈に不適)。URLコピーボタンを設置

### 4.5 SEO・メタ情報

- 各ページに OGP画像(Pillowで生成、1200x630)、`sitemap.xml` 登録、JSON-LD構造化データ
- OGP画像URLにはキャッシュバスター(`?v=N`)を付与できるようにしておく(SNSキャッシュ問題の既知対策)
- ページタイトルには問いかけを含める(例:「人が増えれば早く終わる? | ブルックスの法則シミュレーター」)

### 4.6 デザイン

- QCDシミュレーターのビジュアルトーンを基準に共通デザイントークン(色・フォント・余白)を `common.css` に定義
- レスポンシブ必須(PCでの投影 + スマホ閲覧の両方を想定)
- 操作系はスライダー + 数値入力の併用を基本とする(QCDのドラッグ操作の簡易版)

## 5. ランディングページ仕様(案)

- シミュレーターをカード形式で一覧表示
- カード要素: 名称(問い)、サムネイル(OGP流用可)、対象読者タグ、公開/準備中バッジ
- 準備中もカードとして掲載し、シリーズの広がりを見せる(クリック不可 or 「準備中」表示)
- 検討済みの将来テーマ: 技術的負債の複利、修正コストの段階的増加、見積もりの不確実性、コードレビュー係数、WIP制限とリードタイム、リリース頻度

## 6. 初期実装シミュレーター仕様

### 6.1 ブルックスの法則(brooks-law)

- **問い**: 「人が増えれば早く終わる?」
- **可視化**: 人数スライダー(2〜20人)に応じて、メンバーノードとコミュニケーションパスを線で描画、パス数 n(n-1)/2 をカウンター表示
- **モデル**: 実効生産性 = 人数 × 個人生産性 − パス数 × コミュニケーションコスト係数。係数はスライダーで調整可能にし、増員で総生産性が下がり始める最適分岐点を強調表示
- **発展要素(任意)**: 途中参加メンバーの立ち上げ時間(ランプアップコスト)のトグル
- **対策・アクション**: チーム分割(疎結合)、インターフェースの明確化、増員は早期に限るなど
- **紹介書籍**: 「人月の神話」

### 6.2 稼働率100%の罠(utilization-trap)

- **問い**: 「全員が常に忙しい=効率的?」
- **可視化**: 稼働率スライダー(50%〜99%)に対する待ち時間を ρ/(1−ρ) で非線形に増えるグラフ(M/M/1近似)。90%を超えると曲線が急上昇する様子を体験させる
- **補助表現**: タスクが順に並ぶ簡易アニメーション(キュー詰まりの直感化)があると効果的
- **対策・アクション**: 意図的なスラック確保、WIP制限、割り込みレーンの設計
- **紹介書籍**: 「ザ・ゴール」「ザ・ボトルネック」「エッセンシャル思考」など

### 6.3 コンテキストスイッチ(context-switch・任意)

- **問い**: 「3案件並行なら1/3ずつ進む?」
- **可視化**: 並行プロジェクト数(1〜5)に応じて、実効作業時間が切り替えロスで減っていく積み上げ棒グラフ(Weinbergのロスモデル: 2件で20%、3件で40%…、初期値にも係数調整可)
- **対策・アクション**: 専任化、会議の集約、集中時間のブロック

## 7. QCDシミュレーター移設手順

1. 既存 `qcd-tradeoff` リポジトリの全内容を統合リポジトリの `qcd-tradeoff/` 配下へコピー
2. 相対パスの調整(assetsを共通化する場合はリンク修正、初回は自己完結のまま移設し、共通化は後続タスクでも可)
3. sitemap.xml・JSON-LD・OGPのURLを新URLに更新
4. **旧リポジトリ側**: 各ページに `<meta http-equiv="refresh">` + canonical + 案内リンクにより、新リダイレクトページに差し替え(GitHub Pagesはサーバーサイド301不可のためこの方式)
5. Search Console等を使っている場合はアドレス変更を申請(未使用なら省略)
6. SNSシェア済みURLのOGPはFacebook Sharing Debuggerで再スクレイプ

## 8. 実装順序(推奨マイルストーン)

1. ✅ リポジトリ作成、共通基盤(`common.css` / `common.js` / i18n枠組み)、ランディングページ骨格
2. ✅ ブルックスの法則シミュレーター(共通基盤の実証を兼ねる)
3. ✅ 稼働率100%の罠シミュレーター
4. ✅ QCD移設 + 旧URLリダイレクト(共通chromeへのリファクタも同時実施)
5. ✅ OGP画像生成、sitemap/robots/JSON-LD整備(全ページ対応済み)
6. ✅ コンテキストスイッチ(初回リリースに含める判断で実装済み)
7. 公開 → クチコミ/Xで反応を見て次テーマを決定(5節の将来テーマ候補から選定)

## 9. 検証ルール(検証ファースト、進行)

各変更を成果物に反映する前に必ず実施:

- HTML/JSの構文チェック(`node --check` 相当、またはブラウザコンソールでエラーゼロ確認)
- **i18n辞書キーの差分確認**: ja と en(および DEV オーバーレイ)のキー構造が一致しているか、機械的に照合するスクリプトを用意する
- シミュレーターの計算ロジックは純関数に切り出し、簡易ユニットテスト(Node実行可能な形)で境界値を検証(例: パス数 n=2→1, n=10→45 / ρ=0.99で発散に近づく等)
- 日英両言語での表示確認、モバイル幅での表示確認
